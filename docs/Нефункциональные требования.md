# Нефункциональные требования

## Интерактивный музейный стенд ГрГУ

**Проект:** `museum` (monorepo full-stack TypeScript)  
**Версия документа:** 1.0  
**Основание:** исходный код репозитория, конфигурация инфраструктуры, архитектурная документация проекта  
**Статус:** описывает фактически реализованные и проектно обусловленные нефункциональные свойства системы

---

## Введение

Нефункциональные требования определяют качественные характеристики программного комплекса `museum`, не сводимые к перечню отдельных функций. Объектом эксплуатации является **интерактивный музейный стенд** — SPA-приложение в полноэкранном режиме браузера на сенсорном киоске или выставочном компьютере в здании Гродненского государственного университета имени Янки Купалы.

Архитектура системы (React 19 + Vite 8, Express 5, PostgreSQL, Prisma, Better Auth), режим работы (локальный киоск, ограниченное число одновременных пользователей) и характер контента (мультимedia, PDF-flipbook, блочная CMS) определяют специфику нефункциональных требований, изложенных ниже.

**Обозначения:** NFT-{категория}-{номер}, где категории: **П** — производительность, **Б** — безопасность, **Н** — надёжность, **М** — масштабируемость, **Д** — доступность, **У** — удобство использования, **А** — адаптивность, **С** — сопровождаемость, **О** — отказоустойчивость.

## 1. Требования к производительности

| ID | Требование | Обоснование (особенности проекта) |
|----|------------|-----------------------------------|
| NFT-П-01 | Клиентское приложение должно обеспечивать отзывчивый интерфейс на киоске за счёт SPA-архитектуры без полной перезагрузки страницы при навигации между разделами | Стенд работает как одностраничное приложение (React Router DOM 7); переходы между разделами выполняются на клиенте, что критично для непрерывного взаимодействия посетителя с тач-экраном |
| NFT-П-02 | Production-сборка frontend должна выполняться через Vite с минификацией и разделением vendor-chunks | `@museum/web` использует Vite 8; автоматическое code splitting снижает объём начальной загрузки JavaScript на киоске |
| NFT-П-03 | Статические медиафайлы (изображения, видео, документы) должны раздаваться напрямую Express-сервером из `apps/web/public/` без прохождения через бизнес-логику API | Медиатека стенда может содержать сотни файлов; прямой доступ по URL `/images`, `/videos`, `/files` снижает нагрузку на API и ускоряет отображение галерей |
| NFT-П-04 | Повторное открытие PDF-книги на страницах памяти должно выполняться быстрее за счёт клиентского кэша | Рендеринг PDF через `pdfjs-dist` ресурсоёмок; `MemoryWarPage` сохраняет растрированные страницы в IndexedDB (`book-cache`), что сокращает время повторного просмотра архивного документа |
| NFT-П-05 | Загрузка файлов в админ-панели ограничена 50 МБ на файл и 20 файлами за один запрос (multer) | Предотвращает блокировку сервера при загрузке крупных видео или PDF; ограничение задано в `media.router.ts` и соответствует профилю нагрузки музейного стенда с единичным редактором контента |
| NFT-П-06 | Анимации интерфейса должны иметь короткую длительность (0,18–0,7 с) и не блокировать взаимодействие | Framer Motion используется на стенде (ScreenSaver, галереи, flipbook); короткие анимации соответствуют сценарию киоска, где посетитель ограничен по времени и не должен ожидать завершения декоративных эффектов |
| NFT-П-07 | Запросы данных при смене URL должны отменяться при размонтировании компонента (флаг `cancelled` в hooks) | Быстрая навигация по меню стенда создаёт race condition; паттерн в `usePageByPath`, `useMenuSection` предотвращает некорректное отображение контента предыдущего раздела |
| NFT-П-08 | Изображения в фотогалерее должны загружаться с атрибутом `loading="lazy"` | `PhotoGallery` содержит большое число снимков, сгруппированных по годам; отложенная загрузка снижает начальный объём сетевого трафика при открытии галереи |
| NFT-П-09 | Поиск людей в админ-панели должен выполняться с debounce 300 ms | `PeoplePanel` отправляет запрос `GET /api/people?q=…` только после паузы ввода, что снижает число обращений к PostgreSQL при наборе текста администратором |

## 2. Требования к безопасности

| ID | Требование | Обоснование (особенности проекта) |
|----|------------|-----------------------------------|
| NFT-Б-01 | Доступ к административной панели и мутирующим API-эндпоинтам должен быть возможен только после аутентификации | Стенд публичен, но CMS, медиа и справочники изменяются только авторизованными пользователями; middleware `requireAuth` защищает `POST`/`PUT`/`PATCH`/`DELETE` маршруты |
| NFT-Б-02 | Аутентификация должна выполняться через Better Auth с email/password и сессионными HttpOnly-cookie | Пароли не передаются в localStorage; cookie с флагами `httpOnly`, `sameSite: 'lax'`, `secure` (в production) снижают риск XSS-эксfilтрации сессии на общедоступном киоске |
| NFT-Б-03 | Публичная саморегистрация пользователей должна быть отключена (`disableSignUp: true`) | Музейный стенд не предполагает пользовательских аккаунтов посетителей; учётные записи администраторов создаются только через `npm run db:seed-admin` |
| NFT-Б-04 | Минимальная длина пароля администратора — 12 символов (Better Auth + HTML5 `minLength` в `AdminLogin`) | Ограничение согласовано между backend и frontend; повышает стойкость к подбору пароля при небольшом числе администраторов |
| NFT-Б-05 | В production запуск сервера должен быть заблокирован при использовании dev-секрета Better Auth | Функция `assertProductionSecrets()` проверяет, что `BETTER_AUTH_SECRET` изменён; предотвращает развёртывание с известным ключом подписи сессий |
| NFT-Б-06 | На endpoint аутентификации `/api/auth` должно действовать ограничение частоты запросов (20 req/min) | `express-rate-limit` защищает от brute-force атак на форму входа `/admin/login`; актуально при размещении админки в той же сети, что и киоск |
| NFT-Б-07 | HTTP-ответы должны содержать защитные заголовки через middleware `helmet` | Снижает риск clickjacking и ряда типовых web-уязвимостей при работе стенда в браузере с полным доступом к DOM |
| NFT-Б-08 | CORS должен быть настроен на явный список origin из `CORS_ORIGIN` с поддержкой `credentials: true` | Frontend и API могут работать на разных портах в dev (Vite :5173, Express :3000); whitelist origin предотвращает несанкционированные cross-origin запросы с cookie |
| NFT-Б-09 | Клиентский код не должен содержать секретов; конфигурация через `VITE_*` — только для публичных URL | `.env.example` не содержит секретов frontend; `BETTER_AUTH_SECRET` доступен только на сервере |
| NFT-Б-10 | Текстовый контент CMS должен экранироваться React по умолчанию; HTML-блоки `richText` требуют санитизации при расширении | React JSX предотвращает базовый XSS; при хранении HTML в JSONB-документах CMS риск сохраняется — актуально для админки, доступной доверенным редакторам музея |
| NFT-Б-11 | Загрузка и изменение медиафайлов доступны только после аутентификации | Предотвращает анонимную подмену экспозиционного контента на публичном стенде ГрГУ |

## 3. Требования к надёжности

| ID | Требование | Обоснование (особенности проекта) |
|----|------------|-----------------------------------|
| NFT-Н-01 | Данные контента должны храниться в PostgreSQL с транзакционной целостностью | Страницы CMS, персоналии, меню и метаданные медиа связаны реляционными ключами; Prisma ORM обеспечивает ACID-гарантии при публикации и редактировании |
| NFT-Н-02 | CMS-страницы должны поддерживать разделение черновика (`draftDocument`) и опубликованной версии (`publishedDocument`) | Редактирование в админке не должно немедленно влиять на киоск; посетитель всегда видит только `publishedDocument` через public API |
| NFT-Н-03 | При сохранении черновика CMS должна применяться проверка версии документа (optimistic locking) | Поле `documentVersion` инкрементируется при каждом сохранении; расхождение версий возвращает HTTP 409, предотвращая тихую перезапись при параллельном редактировании |
| NFT-Н-04 | Каждая публикация CMS-страницы должна создавать запись в `page_versions` | История версий позволяет восстановить контент после ошибочного редактирования — критично для музейной экспозиции с историческими материалами |
| NFT-Н-05 | Удаление страниц, персон и медиа-записей должно выполняться как soft delete (`deletedAt`) | Снижает риск безвозвратной потери экспозиционных данных при ошибке администратора |
| NFT-Н-06 | При изменении slug CMS-страницы должен создаваться редирект в `page_redirects` | Закладки и QR-коды на стенде могут ссылаться на старые URL; `resolveSlug()` прозрачно перенаправляет запросы |
| NFT-Н-07 | Схема БД должна управляться версионированными миграциями Prisma | Команды `db:migrate`, `db:deploy` обеспечивают воспроизводимость структуры БД при развёртывании на сервере университета |
| NFT-Н-08 | Серверные ошибки должны возвращать структурированный JSON `{ error: string }` через централизованный `errorHandler` | Единый формат ошибок упрощает обработку на клиенте (`ApiError`) и диагностику сбоев |
| NFT-Н-09 | Публикация CMS-страницы должна выполняться в транзакции Prisma (`$transaction`) | Атомарность обновления `publishedDocument` и создания `pageVersion` исключает рассогласование данных |

## 4. Требования к масштабируемости

| ID | Требование | Обоснование (особенности проекта) |
|----|------------|-----------------------------------|
| NFT-М-01 | Добавление новых разделов экспозиции должно быть возможно без изменения исходного кода frontend | Catch-all маршрут `*` + `PathResolverPage` + динамическое меню из `menu_items` позволяют создавать разделы через админку (menu + CMS) |
| NFT-М-02 | Типы CMS-блоков должны расширяться через реестр без миграции схемы БД | Контент хранится в JSONB; новый блок добавляется в `cms-block-registry.ts`, `BlockRenderer` и `BlockEditor` — модель данных не меняется |
| NFT-М-03 | Классификация персоналий должна расширяться через справочники ролей, тегов и категорий | Музей ГрГУ охватывает разные тематики (ректоры, спорт, память); CRUD справочников в `TaxonomyPanel` не требует изменения кода страниц |
| NFT-М-04 | Monorepo-структура должна поддерживать независимое развитие frontend, backend и shared-пакета | `@museum/document` содержит общие типы CMS; изменения контракта блоков синхронизируются между редактором и рендерером через один пакет |
| NFT-М-05 | Turborepo должен orchestrate сборку с учётом зависимостей пакетов | `@museum/server#build` зависит от `@museum/web#build`; гарантирует, что production-сервер раздаёт актуальную статику |
| NFT-М-06 | Система должна быть рассчитана на профиль нагрузки «один киоск + единичные администраторы», а не на массовый публичный веб-трафик | Архитектура SPA без SSR, CDN и горизонтального масштабирования соответствует размещению на сервере университета с ограниченной аудиторией |
| NFT-М-07 | Внешние видеоресурсы (YouTube, Vimeo) должны подключаться через регистрацию ссылки без локального хранения | `upload-url` и `is_external` в `media_assets` позволяют расширять видеогалерею без роста дискового пространства сервера |

## 5. Требования к доступности (accessibility)

| ID | Требование | Обоснование (особенности проекта) |
|----|------------|-----------------------------------|
| NFT-Д-01 | Формы админ-панели должны использовать семантические элементы `<form>`, `<label>`, связанные с полями ввода | `AdminLogin`, `PersonForm`, панели CMS соответствуют базовым требованиям HTML-форм; необходимо для работы со screen reader при редактировании контента |
| NFT-Д-02 | Toast-уведомления в админке должны иметь `aria-live="polite"` и `role="status"` | `AdminToastContext` обеспечивает озвучивание результата операций сохранения/ошибки ассистивными технологиями |
| NFT-Д-03 | Интерактивные элементы админки должны иметь `aria-label` на icon-only кнопках | Drag-handle, закрытие toast, действия в `PeoplePanel` — частичное покрытие ARIA в административной части |
| NFT-Д-04 | Поля ввода админки должны иметь видимое состояние focus (`focus:ring-2`) | `adminInputClass` и форма входа обеспечивают визуальную индикацию фокуса при клавиатурной навигации |
| NFT-Д-05 | Публичный стенд в текущей версии имеет ограниченную поддержку accessibility: отсутствуют skip links, `lang="ru"` в `index.html` (установлен `lang="en"`), малое число ARIA-атрибутов на киоск-кнопках | Киоск ориентирован преимущественно на touch-взаимодействие; документированное ограничение зафиксировано для последующей доработки |
| NFT-Д-06 | Модальные окна должны поддерживать закрытие по клавише Escape (видеогалерея) | `VideoModal` регистрирует обработчик `keydown`; частичная клавиатурная доступность для модальных сценариев |
| NFT-Д-07 | Изображения в CMS и медиатеке должны поддерживать поле `alt` для текстовой альтернативы | Модель `MediaAsset.alt` и метаданные галереи; актуально для посетителей с нарушениями зрения при расширении a11y |

## 6. Требования к удобству использования (usability)

| ID | Требование | Обоснование (особенности проекта) |
|----|------------|-----------------------------------|
| NFT-У-01 | Элементы управления на главном экране должны иметь крупный touch target (кнопки 320×144 px) | `Home.tsx` проектируется под сенсорный киоск; крупные кнопки снижают ошибки нажатия в условиях выставочного зала |
| NFT-У-02 | Навигация «назад» должна быть доступна тремя способами: кнопка в navbar, жест свайпа вправо, кнопка «Назад» на экране подменю | Дублирование снижает когнитивную нагрузку; посетитель не «застревает» в глубоком разделе экспозиции |
| NFT-У-03 | После 5 минут бездействия стенд должен возвращаться на главный экран через ScreenSaver | Киоск используется последовательно разными посетителями; сброс контекста предотвращает демонстрацию «чужой» сессии просмотра |
| NFT-У-04 | При нажатии кнопок должен обеспечиваться тактильный отклик (`active:scale-95`) | Визуальная обратная связь подтверждает регистрацию касания на touch-экране без звукового сопровождения |
| NFT-У-05 | Состояния загрузки, ошибки и пустого контента должны отображаться явно (`LoadingState`, `ErrorState`, `EmptyState`) | Посетитель киоска не должен интерпретировать белый экран как зависание; компоненты `States.tsx` используются на всех data-driven страницах |
| NFT-У-06 | Админ-панель должна предоставлять toast-уведомления об успехе и ошибке каждой операции | Редактор музея не обязан быть техническим специалистом; немедленная обратная связь снижает неопределённость при публикации контента |
| NFT-У-07 | CMS-редактор должен поддерживать WYSIWYG-подобный preview через `BlockRenderer` | Администратор видит результат до публикации на стенде; снижает число итераций «опубликовал — проверил — исправил» |
| NFT-У-08 | Глубина навигации до контента не должна превышать 2–4 перехода от главного экрана | Иерархия «Home → подменю → CMS/спецстраница» соответствует ограниченному времени внимания посетителя музея |
| NFT-У-09 | Тексты ошибок API должны быть человекочитаемыми (на русском языке) | `ApiMessage` на сервере и обработка `ApiError` на клиенте; администратор получает понятные сообщения («Страница не найдена», «Версия документа устарела») |
| NFT-У-10 | ScreenSaver должен содержать идентификацию учреждения (ГрГУ имени Янки Купалы) и призыв «Нажмите, чтобы продолжить» | Заставка выполняет функцию ориентации и приглашения к взаимодействию в режиме ожидания киоска |

## 7. Требования к адаптивности интерфейса

| ID | Требование | Обоснование (особенности проекта) |
|----|------------|-----------------------------------|
| NFT-А-01 | Основным целевым устройством является большой сенсорный экран (киоск) с разрешением desktop-класса | Layout использует `w-screen h-screen`, крупные кнопки Home; mobile-first подход не является доминирующим |
| NFT-А-02 | Viewport должен быть настроен для корректного масштабирования (`width=device-width`) | `index.html` содержит meta viewport; необходимо для работы на планшетах и альтернативных тач-устройствах |
| NFT-А-03 | CMS-блоки должны поддерживать responsive-раскладку (breakpoints `md:`, `lg:`) | `CmsBlockViews.tsx` адаптирует сетки галерей и двухколоночные блоки; контент CMS — наиболее адаптивная часть стенда |
| NFT-А-04 | Страница ректоров (`Rectors`) адаптивна: на экранах ≥ `md` — чередующийся timeline по центру; на узких — вертикальная линия с точками слева и карточками справа (CSS Grid) | Оба варианта сохраняют хронологию с фото, ФИО и годами; подсказка «листайте» скрывается у нижней границы списка |
| NFT-А-05 | Админ-панель использует фиксированный sidebar 256 px (`w-64`) и рассчитана преимущественно на desktop | Редактирование контента выполняется с рабочего места сотрудника музея, а не с киоска |
| NFT-А-06 | Главный экран использует `flex-wrap` для кнопок разделов с ограничением `max-w` контейнера | При большом числе пунктов меню `home` кнопки переносятся на следующую строку без выхода за экран |
| NFT-А-07 | Карточка ректора (`RectorDetails`) использует `flex-col md:flex-row` для адаптации блока с фото и текстом | Частичная адаптивность детальных страниц персоналий |

## 8. Требования к сопровождаемости

| ID | Требование | Обоснование (особенности проекта) |
|----|------------|-----------------------------------|
| NFT-С-01 | Весь код monorepo должен быть написан на TypeScript (~6.0) с включённой проверкой типов | `npm run type-check` через Turborepo; типобезопасность критична для CMS-редактора с 22 типами блоков |
| NFT-С-02 | Контракт CMS-документов должен быть вынесен в shared-пакет `@museum/document` | Единые типы `PageDocument`, `BlockNode` для frontend и backend исключают расхождение формата JSONB |
| NFT-С-03 | Backend должен следовать слоистой архитектуре: routes → controllers → services → Prisma | Модули `people`, `pages`, `menu`, `media` изолированы; упрощает локализацию изменений при расширении API |
| NFT-С-04 | Frontend должен следовать компонентной структуре: pages / layouts / patterns / cms / design-system / features | ~100+ файлов в `apps/web/src`; разделение обеспечивает навигацию по кодовой базе при доработке UI стенда |
| NFT-С-05 | Расширение CMS-блока должно требовать изменений в трёх точках: registry, view (`CmsBlockViews`), editor (`BlockEditor`) | Документированный паттерн; новый тип блока не затрагивает остальные 20 типов |
| NFT-С-06 | Качество кода должно контролироваться ESLint и Prettier (`npm run lint`, `npm run check`) | Единый стиль и статический анализ в monorepo; снижает стоимость code review при курсовой/дипломной разработке |
| NFT-С-07 | Конфигурация окружения должна быть вынесена в `.env` с примером `.env.example` | `PORT`, `DATABASE_URL`, `BETTER_AUTH_SECRET`, `CORS_ORIGIN`, `ADMIN_EMAIL` — развёртывание на сервере ГрГУ без правки исходного кода |
| NFT-С-08 | Схема БД должна описываться декларативно в `schema.prisma` с генерацией клиента | Prisma Migrate и `db:generate` автоматизируют синхронизацию ORM и PostgreSQL |
| NFT-С-09 | README должен содержать инструкции по установке, разработке, сборке и миграциям | `npm install`, `npm run dev`, `npm run build`, `db:migrate`, `db:seed-admin` — минимальный набор для передачи проекта следующему разработчику |
| NFT-С-10 | Фиксированные React-маршруты (spec-экраны) и catch-all CMS (`PathResolverPage`) сосуществуют: hub + CMS для типовых разделов, spec-страницы — для уникального UX | Снижает необходимость создания нового `.tsx`-файла для каждого URL при типовой экспозиции |

## 9. Требования к отказоустойчивости

| ID | Требование | Обоснование (особенности проекта) |
|----|------------|-----------------------------------|
| NFT-О-01 | При ошибке загрузки API меню `PathResolverPage` должен выполнять fallback на CMS-страницу по текущему пути | Сбой навигационного API не должен блокировать доступ к контенту; посетитель получает CMS-страницу или empty state |
| NFT-О-02 | При ошибке загрузки изображения должен отображаться placeholder или элемент скрывается (`onError`) | Битые ссылки на медиа в архиве не должны «ломать» layout галереи и карточек ректоров |
| NFT-О-03 | При HTTP 401 в админке клиент должен перенаправлять на `/admin/login` | Истечение сессии не оставляет администратора в нерабочем состоянии интерфейса |
| NFT-О-04 | Необработанные серверные исключения должны возвращать HTTP 500 с общим сообщением без раскрытия stack trace | `errorHandler` логирует ошибку в console, клиент получает `{ error: "Internal error" }`; предотвращает утечку внутренней информации на киоске |
| NFT-О-05 | При конфликте версий CMS-клиент должен отображать toast с ошибкой, не перезаписывая данные молча | HTTP 409 (`DOCUMENT_VERSION_STALE`) обрабатывается в `PagesPanel`; администратор осведомлён о необходимости перезагрузки |
| NFT-О-06 | При недоступности PDF-файла flipbook должен отображать понятное сообщение об ошибке | `MemoryWarPage` показывает экран «Не удалось загрузить PDF» вместо падения компонента |
| NFT-О-07 | При отсутствии опубликованной CMS-страницы должен отображаться empty state, а не HTTP-ошибка в UI | `CmsPageContent` перехватывает 404 API и показывает «Для этого пути страница пока не настроена в CMS» |
| NFT-О-08 | При отсутствии персоны на странице ректора отображается сообщение «Ректор не найден» | `RectorDetails` обрабатывает `error` и `null` из `usePerson` без crash приложения |
| NFT-О-09 | Запросы fetch в hooks должны игнорировать результаты после unmount (`cancelled = true`) | Предотвращает setState на размонтированном компоненте при быстрой навигации — типичный сценарий на киоске |
| NFT-О-10 | Production bootstrap должен проверять подключение к PostgreSQL (`prisma.$connect()`) при старте | Раннее обнаружение недоступности БД до приёма HTTP-запросов от киоска |
| NFT-О-11 | Система не реализует offline/PWA-режим и резервирование серверных компонентов | При обрыве сети киоск теряет доступ к API и CMS-контенту; ограничение зафиксировано — для музейного стенда в локальной сети университета риск считается приемлемым |
