# Аргументация выбора технологий

## Раздел курсовой работы

**Объект:** программный комплекс `museum` — интерактивный музейный стенд Гродненского государственного университета имени Янки Купалы  
**Предмет:** обоснование технологического стека full-stack TypeScript-приложения  
**Основание:** архитектура репозитория, функциональные и нефункциональные требования проекта

---

## 1. Введение

Выбор технологий для музейного стенда ГрГУ определяется не абстрактной «модностью» стека, а совокупностью эксплуатационных и предметных ограничений. Система должна:

- работать как **сенсорный киоск** в полноэкранном браузере с touch-навигацией, ScreenSaver и PDF-flipbook;
- предоставлять **блочную CMS** с 22 типами блоков, версионированием и публикацией без участия программиста;
- хранить **разнородный контент** — персоналии, меню, JSONB-документы, медиафайлы — в единой БД;
- обеспечивать **защищённую админ-панель** для сотрудников музея;
- развёртываться **на сервере университета** без обязательной публичной доступности в интернет.

Исходя из этого профиля, ниже приведена аргументация по каждой **фактически используемой** технологии. Технологии, отсутствующие в репозитории (Next.js, NestJS, Docker, Nginx, Drizzle ORM), рассматриваются в качестве альтернатив там, где это необходимо для обоснования выбора.

## 2. Платформа и организация кодовой базы

### 2.1. Node.js 20

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | Deno, Bun; серверные решения на Python (Django/FastAPI), PHP, Java (Spring) |
| **Преимущества выбранного варианта** | Единый язык TypeScript на клиенте и сервере; асинхронная модель I/O для REST API и загрузки файлов; npm-экосистема с готовыми пакетами (Express, Prisma, Better Auth, pdfjs); LTS-версия 20 зафиксирована в `.nvmrc` |
| **Соответствие требованиям проекта** | Monorepo `@museum/web` + `@museum/server` + `@museum/document` логично собирается на одном runtime; README описывает запуск `npm run dev` / `npm run start` без дополнительных интерпретаторов |
| **Влияние на производительность и сопровождение** | Event loop достаточен для профиля «один киоск + 1–2 администратора»; для сопровождения курсовой работы единый стек снижает порог входа — не требуется переключение между Python-backend и JS-frontend |

Node.js выбран не как «универсальный стандарт», а потому что музейный стенд уже построен на React, а CMS-редактор, flipbook и DnD-библиотеки существуют только в JavaScript-экосистеме. Перенос backend на другой язык не дал бы выигрыша при текущем масштабе API (четыре доменных модуля: people, pages, menu, media).

### 2.2. Monorepo (npm workspaces + Turborepo)

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | Два отдельных репозитория (frontend/backend); Nx; Lerna; pnpm workspaces без orchestrator |
| **Преимущества выбранного варианта** | Turborepo задаёт порядок сборки (`@museum/server#build` зависит от `@museum/web#build`); параллельный `dev` для Vite и Express; общий `npm run check` (lint + type-check) |
| **Соответствие требованиям проекта** | Пакет `@museum/document` содержит типы `PageDocument` и `BlockNode`, импортируемые и клиентом (`BlockRenderer`, `DocumentEditor`), и сервером (валидация JSONB). Без monorepo контракт CMS пришлось бы дублировать или публиковать отдельным npm-пакетом |
| **Влияние на производительность и сопровождение** | Кэш сборки Turbo ускоряет повторные `build`; при добавлении нового типа CMS-блока изменения вносятся в registry, view и editor — TypeScript сразу показывает расхождения между пакетами |

### 2.3. TypeScript (~6.0)

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | JavaScript без типов; JSDoc-типизация; Flow |
| **Преимущества выбранного варианта** | Статическая проверка props в 22 типах CMS-блоков; типобезопасные Prisma-модели; общий пакет `@museum/document`; ESLint + `tsc --noEmit` в CI-подобном скрипте `npm run check` |
| **Соответствие требованиям проекта** | CMS-документ — JSONB в PostgreSQL, но структура блоков жёстко типизирована в коде (`BlockType`, `cms-block-registry.ts`). Без TypeScript ошибка в payload блока (например, отсутствие поля `columns` у `imageGallery`) обнаружилась бы только в runtime на киоске |
| **Влияние на производительность и сопровождение** | Нулевой runtime-overhead (компиляция в JS); сопровождение: рефакторинг API (`PersonDto`, `DraftPage`) безопасен — IDE и компилятор отлавливают несовместимые изменения между admin UI и REST endpoints |

## 3. Frontend

### 3.1. React 19

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | Vue 3, Svelte, Angular, Preact |
| **Преимущества выбранного варианта** | Компонентная модель естественно ложится на dual-UI: киоск (`Home`, `BlockRenderer`, `ScreenSaver`) и админка (`AdminPanel`, `DocumentEditor`); экосистема библиотек для touch (Framer Motion), DnD (`@dnd-kit`), PDF (`pdfjs-dist`, `react-pageflip`) |
| **Соответствие требованиям проекта** | `BlockRenderer` рекурсивно рендерит дерево блоков CMS — декларативный JSX проще, чем шаблоны с императивным DOM. Специализированные экраны (timeline ректоров, `EntityListDetail` на страницах памяти) реализованы как изолированные React-компоненты; блок `peopleCatalog` — для картотек на CMS-страницах |
| **Влияние на производительность и сопровождение** | Virtual DOM достаточен для киоска: навигация между разделами без полной перезагрузки страницы. Структура `pages/ → patterns/ → design-system/ → cms/` облегчает добавление нового экрана (например, ещё одной тематической галереи) без переписывания каркаса |

Vue или Svelte могли бы дать меньший bundle, но потребовали бы замены всей экосystem-зависимости (react-pageflip, better-auth/react, @dnd-kit). Для проекта с уже реализованным UI на React миграция не оправдана.

### 3.2. Vite 8 (вместо Next.js)

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | **Next.js** (SSR/SSG), Create React App, Webpack вручную, Parcel |
| **Преимущества выбранного варианта** | Быстрый HMR при разработке CMS-редактора; ESM-native dev server; proxy `/api` → Express в `vite.config.ts`; production build без overhead SSR-рантайма |
| **Соответствие требованиям проекта** | Музейный стенд — **локальный киоск**, SEO и индексация поисковиками не входят в требования. SSR Next.js не нужен: посетитель всегда получает приложение из локальной сети университета. Catch-all маршрут `*` и `PathResolverPage` реализуют динамическую навигацию на клиенте — аналог файловой маршрутизации Next.js, но через React Router и меню из БД |
| **Влияние на производительность и сопровождение** | SPA загружает `index.html` один раз; последующие переходы — client-side routing (быстро на киоске). Сопровождение: один конфиг Vite + React Router проще, чем App Router + Server Components + API Routes Next.js для команды, которая уже вынесла API в Express |

Next.js был бы оправдан для **публичного сайта** с SEO (как в отдельном документе `PRODUCT_DOCS.md`, не относящемся к `museum`). Для киоска с PDF-flipbook, Framer Motion и `@dnd-kit` в CMS-редакторе client-only архитектура снижает сложность деплоя: Express раздаёт `apps/web/dist` без Node.js SSR-слоя.

### 3.3. React Router DOM 7

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | TanStack Router, Wouter; маршрутизация Next.js; state-based «виртуальные страницы» без URL |
| **Преимущества выбранного варианта** | Декларативные маршруты в `routes.tsx`; `ProtectedRoute` для `/admin`; catch-all `*` для CMS-путей; поддержка params (`/history/rectors/:id`) |
| **Соответствие требованиям проекта** | Часть разделов — **фиксированные маршруты** (`/gallery`, `/history/memory/vov`), часть — **динамические** (любой slug из `menu_items` + CMS). React Router совмещает оба подхода: явные Route для спецэкранов и `PathResolverPage` для остального. ScreenSaver и swipe-back работают поверх history API |
| **Влияние на производительность и сопровождение** | URL сохраняют контекст (администратор может открыть `/admin` после логина с redirect); для киоска deep links второстепенны, но помогают при отладке. Добавление нового спецэкрана — одна строка в `routes.tsx` |

### 3.4. Tailwind CSS 4

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | CSS Modules, styled-components, MUI/Chakra UI, Bootstrap, plain CSS |
| **Преимущества выбранного варианта** | Utility-классы для быстрой вёрстки touch-кнопок (`w-80 h-36`, `active:scale-95`); `@tailwindcss/vite` без отдельного PostCSS-конфига; glassmorphism (`bg-white/70 backdrop-blur`) повторяется на Home и MainLayout |
| **Соответствие требованиям проекта** | Киоск требует **крупных, предсказуемых** элементов управления — Tailwind позволяет задать фиксированные размеры кнопок и отступы без создания сотен CSS-файлов. Админка использует отдельные классы (`adminInputClass`), но в той же utility-парадигме |
| **Влияние на производительность и сопровождение** | Purge неиспользуемых классов при production build уменьшает CSS; сопровождение: изменение палитры «университетского синего» затрагивает class maps в `Button.tsx`, а не глобальные cascade-правила. Компромисс: два параллельных UI-слоя (kiosk vs admin) — осознанное решение, не следствие выбора Tailwind |

## 4. Backend

### 4.1. Express 5 (вместо NestJS)

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | **NestJS**, Fastify, Koa, Hono; serverless functions |
| **Преимущества выбранного варианта** | Минимальный каркас: `createApp()` → middleware → routers → static files; прямое монтирование Better Auth (`toNodeHandler`); низкий порог входа для курсовой разработки |
| **Соответствие требованиям проекта** | API музейного стенда — **четыре REST-модуля** и auth. NestJS добавил бы модули, декораторы, DI-контейнер и boilerplate для guards, тогда как `requireAuth` — один middleware-файл. Express в production **одновременно** отдаёт API, SPA (`web/dist`) и медиа (`web/public`) — для киоска в локальной сети достаточно одного процесса |
| **Влияние на производительность и сопровождение** | Express 5 на Node 20 обрабатывает нагрузку киоска с запасом. Сопровождение: слой `controllers → services → prisma` повторяет структуру NestJS вручную, но без магии декораторов — проще трассировать запрос от `pages.router.ts` до `PagesService.publish()` |

NestJS оправдан при десятках модулей, микросервисах, WebSocket, очередях. В `museum` нет BullMQ, GraphQL, WebSocket — выбор Express снижает cognitive load при сопровождении музеем ГрГУ.

### 4.2. Вспомогательные middleware (helmet, cors, express-rate-limit, multer)

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | Nginx rate limiting; reverse proxy CORS; ручная обработка multipart |
| **Преимущества выбранного варианта** | `helmet` — HTTP security headers; `cors` с whitelist `CORS_ORIGIN` и credentials для cookie-сессий; rate limit 20 req/min на `/api/auth`; multer memory storage до 50 МБ для загрузки архивных PDF и фото |
| **Соответствие требованиям проекта** | NFT-Б-06, NFT-Б-08 из нефункциональных требований; админка и API могут быть на разных портах в dev (5173/3000), в production — same origin через Express |
| **Влияние на производительность и сопровождение** | Multer в memory подходит для редких загрузок администратором; при росте объёма медиа можно перейти на disk storage без смены фреймворка |

## 5. База данных и ORM

### 5.1. PostgreSQL

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | SQLite (embedded), MySQL/MariaDB, MongoDB, Supabase hosted |
| **Преимущества выбранного варианта** | JSONB для CMS-документов (`draftDocument`, `publishedDocument`); ACID-транзакции при публикации страниц; реляционная модель для `people` ↔ `roles` ↔ `tags`; `contains` + `insensitive` для поиска людей в админке |
| **Соответствие требованиям проекта** | Блочная CMS с 22 типами блоков **не требует ALTER TABLE** при добавлении типа — JSONB в PostgreSQL идеален. Каталог ректоров, преподавателей войны, спортсменов — связи many-to-many (`person_roles`, `person_tags`). Меню (`menu_items`) — иерархия с `parent_id`. Эти модели естественно выражаются в SQL, а не в document-store |
| **Влияние на производительность и сопровождение** | PostgreSQL на сервере университета — стандартный выбор для on-premise. Производительность: для одного киоска запросы `GET /api/menu/home`, `GET /api/pages/by-path` не создают нагрузки; индексы на `slug`, `path` (unique) обеспечивают быстрый lookup. Сопровождение: Prisma Migrate версионирует схему (`npm run db:deploy`) |

MongoDB отпала бы из-за необходимости **одновременно** хранить жёстко связанные сущности (люди + роли) и гибкие CMS-документы — PostgreSQL объединяет оба сценария через JSONB + foreign keys. SQLite не подходит для concurrent access при одновременной работе киоска и администратора.

---

### 5.2. Prisma 7 (вместо Drizzle ORM)

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | **Drizzle ORM**, TypeORM, Sequelize, Knex + raw SQL |
| **Преимущества выбранного варианта** | Декларативная `schema.prisma`; автогенерация типобезопасного клиента в `generated/prisma`; Prisma Migrate; официальный `@better-auth/prisma-adapter`; `$transaction` в `PagesService.publish()` |
| **Соответствие требованиям проекта** | Better Auth **из коробки** интегрирован с Prisma (`prismaAdapter`) — таблицы `user`, `session`, `account` описаны в той же схеме. Drizzle-путь потребовал бы `@better-auth/drizzle-adapter` и отдельную schema-as-code. Optimistic locking CMS (`documentVersion`) реализован через `update` с проверкой версии — Prisma Client делает это прозрачно |
| **Влияние на производительность и сопровождение** | Prisma добавляет overhead по сравнению с raw SQL, но для REST API музея он несущественен. Сопровождение: `prisma migrate dev` + `db:generate` — воспроизводимая схема при передаче проекта следующему разработчику; типы `Person`, `Page` синхронизированы с БД автоматически |

Drizzle легче по runtime, но в проекте уже реализованы миграции, seed-admin, legacy-migrate script под Prisma. Смена ORM на этапе курсовой не даёт предметной выгоды для киоска.

## 6. Аутентификация

### 6.1. Better Auth

| Критерий | Содержание |
|----------|------------|
| **Альтернативы** | Passport.js + JWT вручную; Auth.js (NextAuth); Lucia; custom session store; Keycloak |
| **Преимущества выбранного варианта** | TypeScript-first; session cookie (HttpOnly); `disableSignUp: true` — только seed-admin; `minPasswordLength: 12`; интеграция Express через `toNodeHandler`; клиент `better-auth/react` с `useSession()` |
| **Соответствие требованиям проекта** | На стенде **нет пользователей-посетителей** — auth нужен только для `/admin`. Better Auth закрывает сценарий «email + password + session cookie» без реализации refresh rotation, хранения хешей и таблиц sessions вручную. `ProtectedRoute` и `requireAuth` на API используют одну и ту же сессию |
| **Влияние на производительность и сопровождение** | Session lookup на каждый admin-запрос — приемлемая задержка для редких операций редактора. Сопровождение: конфигурация сосредоточена в `auth/auth.ts` (~25 строк); не нужен отдельный auth-микросервис или Keycloak для одного-двух администраторов музея |

JWT в localStorage был бы **хуже для киоска**: тот же браузер используется посетителями и потенциально администратором; HttpOnly cookie изолирует сессию admin от XSS на публичных страницах.

## 8. Инфраструктура развёртывания (осознанный отказ)

### 8.1. Docker, Nginx, Cloudflare Tunnel — не используются

| Критерий | Содержание |
|----------|------------|
| **Альтернативы (не выбраны)** | Docker Compose (app + postgres); Nginx reverse proxy + TLS; Cloudflare Tunnel для внешнего доступа |
| **Причины отказа в данном проекте** | README описывает прямой запуск `npm run build && npm run start`; киоск и сервер — **в локальной сети ГрГУ**; один Express-процесс раздаёт API + SPA + `/images|videos|files`; публичный internet-доступ не требуется |
| **Соответствие требованиям** | NFT-М-06: профиль нагрузки «один киоск + единичные админы»; медиа в `apps/web/public/` — файловая система, не object storage |
| **Влияние** | **Производительность:** для киоска overhead контейнеризации не нужен. **Сопровождение:** меньше moving parts при сдаче курсовой — Node 20 + PostgreSQL + `.env`. Docker/Nginx остаются **рекомендацией на перспективу** при промышленном деплое, но не частью текущей реализации |

## 10. Заключение

Технологический стек `museum` сформирован под задачу **интерактивного музейного стенда ГрГУ**, а не под универсальный публичный портал. Ключевые архитектурные решения следуют из этого:

1. **SPA (React + Vite + React Router)** вместо Next.js — киоск не нуждается в SSR/SEO; нужны клиентские PDF, DnD и анимации.
2. **Express** вместо NestJS — API из четырёх доменов не оправдывает DI-фреймворк; один процесс раздаёт API и статику.
3. **PostgreSQL + JSONB + Prisma** — реляционные связи для людей и меню плюс гибкие CMS-документы в одной СУБД; Better Auth интегрирован через Prisma adapter.
4. **Better Auth** — минимальный auth-контур для единственной роли (администратор музея).
5. **Monorepo + TypeScript** — общий контракт CMS между редактором, рендерером и API.
6. **Отказ от Docker/Nginx** на текущем этапе — упрощение развёртывания в учебной и университетской среде при сохранении возможности добавить их позже.

Выбранные технологии обеспечивают баланс между **производительностью киоска** (client-side routing, static media, PDF cache), **сопровождаемостью** (единый язык, типы, миграции, модульная структура) и **предметными требованиями** музейной экспозиции ГрГУ (блочная CMS, персоналии, flipbook, touch UX).
