# Преимущества разрабатываемой системы

## Интерактивный музейный стенд ГрГУ

**Объект:** программный комплекс `museum`  
**Версия документа:** 1.0  
**Назначение:** обоснование преимуществ разрабатываемой системы по сравнению с традиционными музейными стендами и существующими цифровыми решениями  
**Основание:** исходный код репозитория, функциональные и нефункциональные требования, анализ аналогов

---

## 1. Введение

Музейные информационные системы исторически эволюционировали от **статических стендов** (текстовые таблички, печатные плакаты, физические витрины с подписями) к **цифровым решениям** — интерактивным киоскам, веб-каталогам и специализированному ПО для учёта коллекций. Каждый из этих подходов решает часть задач, но не обеспечивает в совокупности требования университетского музея ГрГУ: сенсорный стенд с предметно-ориентированным контентом, самостоятельное редактирование материалов сотрудниками и единая платформа управления.

Разрабатываемая система `museum` позиционируется как **целостный программный комплекс**, объединяющий публичный интерактивный стенд и административную панель. Ниже приведён перечень преимуществ по семи ключевым направлениям. Каждое преимущество сопровождается обоснованием с опорой на реализованную функциональность проекта и сопоставлением с типовыми альтернативами: традиционными стендами, универсальными CMS (WordPress, Omeka S), системами учёта коллекций (PastPerfect, CollectionSpace) и конструкторами киосков (Intuiface).

## 2. Удобство использования

### 2.1. Интерфейс, спроектированный под сенсорный киоск

**Преимущество.** Пользовательский интерфейс публичной части изначально ориентирован на touch-взаимодействие в условиях выставочного зала, а не адаптирован «задним числом» с desktop-сайта.

**Обоснование.** На главном экране (`Home.tsx`) кнопки разделов имеют размер **320×144 px** — достаточный touch target для посетителей разного возраста и для работы без мыши. Кнопки снабжены тактильным откликом (`active:scale-95`), подтверждающим регистрацию касания. Layout использует полноэкранный режим (`w-screen h-screen`), что соответствует эксплуатации на выделенном киоске.

Традиционный стенд не предоставляет интерактивности; универсальные веб-платформы (Omeka S, WordPress) используют стандартные размеры ссылок и не включают ScreenSaver, жесты и крупные touch-targets «из коробки». Конструктор Intuiface даёт аналогичные возможности, но требует отдельного проектирования каждой сцены без привязки к доменной модели музея ГрГУ.

### 2.2. Многоканальная навигация «назад»

**Преимущество.** Посетитель не «застревает» в глубоком разделе экспозиции: возврат доступен тремя способами.

**Обоснование.** Реализованы: кнопка «Назад» в `MainLayout`; жест **свайпа вправо** (≥ 80 px) в `App.tsx`; кнопка «← Назад» на экране подменю (`SectionMenuPage`). При отсутствии истории браузера выполняется переход на главную страницу.

На традиционном стенде «возврат» физически означает отход от витрины; в статичном PDF или слайд-шоу на киоске навигация часто линейна. В типовых CMS отсутствует жестовая навигация, критичная для сенсорного экрана без клавиатуры.

### 2.3. Автоматический сброс сеанса (ScreenSaver)

**Преимущество.** Киоск автоматически возвращается в нейтральное состояние после периода бездействия, что удобно для последовательного обслуживания разных посетителей.

**Обоснование.** После **5 минут** без событий `mousedown`, `touchstart`, `keydown`, `mousemove` активируется полноэкранная заставка с логотипом ГрГУ и призывом «Нажмите, чтобы продолжить». Касание экрана переводит на главную страницу. На маршрутах `/admin*` ScreenSaver отключён, чтобы не прерывать работу администратора.

Традиционные стенды не имеют «сеанса»; цифровые киоски на базе браузера без специальной логики часто остаются на последнем открытом экране, демонстрируя «чужую» сессию следующему посетителю.

### 2.4. Предметно-ориентированные экраны вместо универсальных шаблонов

**Преимущество.** Для ключевых тем музея ГрГУ реализованы специализированные интерфейсы, оптимизированные под конкретный тип контента.

**Обоснование.** Система включает:

- **хронологию ректоров** (`/history/rectors`) — timeline с фото, ФИО и годами: на desktop — чередование вокруг центральной линии, на узких экранах — линия слева и карточки справа; подсказка «листайте» скрывается в конце списка;
- **карточку ректора** — развёрнутая биография, галерея, документы;
- **раздел «Купаловцы помнят»** — вкладки «Преподаватели» / «Книга», каталог `EntityListDetail`, PDF-flipbook;
- **фото- и видеогалереи** — группировка по годам, lightbox, фильтр видео по тегам.

Традиционный стенд ограничен статичным текстом и изображениями. Универсальные CMS отображают контент через общие шаблоны; для flipbook, timeline и ролевого каталога преподавателей потребовались бы сторонние плагины или кастомная разработка.

### 2.5. Явная обратная связь при загрузке и ошибках

**Преимущество.** Посетитель и администратор всегда понимают текущее состояние системы; «белый экран» не воспринимается как зависание.

**Обоснование.** На всех data-driven страницах используются компоненты `LoadingState`, `ErrorState`, `EmptyState`. Примеры: «Для этого пути страница пока не настроена в CMS»; «Ректор не найден»; «Не удалось загрузить PDF». В админке — toast-уведомления об успехе и ошибке каждой операции (`AdminToastContext`).

## 3. Скорость доступа к информации

### 3.1. SPA-архитектура без перезагрузки страницы

**Преимущество.** Переход между разделами экспозиции выполняется мгновенно на клиенте, без полной перезагрузки HTML-документа.

**Обоснование.** Приложение построено как **Single Page Application** (React 19 + React Router DOM 7). При навигации по меню, подразделам и CMS-страницам меняется только содержимое маршрута; статические ресурсы и JavaScript загружаются один раз. Это критично для непрерывного взаимодействия посетителя с тач-экраном.

Традиционный стенд не имеет «навигации» в цифровом смысле. Классические веб-сайты на server-side rendering перезагружают страницу при каждом переходе, что на киоске создаёт заметную паузу и мерцание.

### 3.2. Прямая раздача медиафайлов без прохождения через API

**Преимущество.** Изображения, видео и документы доставляются посетителю по кратчайшему пути, без нагрузки на бизнес-логику сервера.

**Обоснование.** В production Express раздаёт статику из `apps/web/public/` по URL `/images`, `/videos`, `/files`. Запрос фотографии в галерее не проходит через REST API и Prisma — браузер получает файл напрямую. Метаданные (подписи, год, теги) загружаются через API один раз; сами файлы — параллельно и с `loading="lazy"` в фотогалерее.

В традиционном стенде скорость «доступа» ограничена физическим перемещением посетителя. В CMS-centric решениях медиа иногда проксируются через backend, что увеличивает задержку при больших коллекциях.

### 3.3. Кэширование PDF-книги на клиенте

**Преимущество.** Повторное открытие архивного документа в разделе памяти выполняется существенно быстрее.

**Обоснование.** Рендеринг PDF через `pdfjs-dist` ресурсоёмок. `MemoryWarPage` сохраняет растрированные страницы в **IndexedDB** (`book-cache`); при повторном открытии flipbook страницы загружаются из локального кэша браузера, а не перерабатываются заново.

Традиционный стенд предлагает только физическую книгу или статичные копии страниц. Типовые CMS не реализуют flipbook с клиентским кэшем «из коробки».

### 3.4. Неглубокая иерархия навигации

**Преимущество.** Любой материал экспозиции доступен за 2–4 перехода от главного экрана.

**Обоснование.** Структура «Главная → подменю секции → CMS-страница / спецэкран» соответствует ограниченному времени внимания посетителя музея. Динамическое меню (`menu_items`) формирует плоскую иерархию внутри секции; catch-all маршрут `PathResolverPage` автоматически выбирает подменю или CMS-контент.

Многоуровневые веб-каталоги (Omeka S, CollectionSpace + custom frontend) часто требуют 4–6 кликов до целевого объекта. Традиционный стенд не страдает от «глубины меню», но и не даёт доступа к сотням материалов в одной точке.

### 3.5. Отмена устаревших запросов при быстрой навигации

**Преимущество.** При быстром переключении разделов посетитель не видит «мигание» контента предыдущего экрана.

**Обоснование.** Hooks `usePageByPath`, `useMenuSection` и аналогичные используют флаг `cancelled` при размонтировании компонента: результат запроса, пришедший после ухода со страницы, игнорируется. Это предотвращает race condition, типичный для SPA с асинхронной загрузкой данных.

## 4. Возможности администрирования

### 4.1. Единая веб-панель для всего контента стенда

**Преимущество.** Сотрудник музея управляет меню, страницами, персоналиями, справочниками и медиафайлами из одного интерфейса `/admin`, без переключения между разрозненными инструментами.

**Обоснование.** Админ-панель включает пять модулей: **Меню разделов**, **CMS страницы**, **Люди**, **Справочники** (роли, теги, категории), **Медиа** (File Manager). Все операции выполняются через REST API с аутентификацией Better Auth.

Традиционный стенд требует физической замены материалов или привлечения дизайнера. Комбинация PastPerfect (учёт) + Intuiface (киоск) + отдельный сайт потребовала бы трёх систем и синхронизации данных между ними.

### 4.2. Блочный CMS-редактор с preview и 22 типами блоков

**Преимущество.** Администратор собирает экспозиционные страницы из готовых блоков без знания HTML и без участия программиста при типовых изменениях.

**Обоснование.** `DocumentEditor` поддерживает добавление, удаление, перетаскивание и редактирование блоков: `hero`, `richText`, `imageGallery`, `video`, `tabs`, `timeline`, `peopleCatalog`, `accordion` и др. Режим **preview** рендерит черновик через тот же `BlockRenderer` (с флагом `preview`), что и публичный стенд — WYSIWYG-подобный опыт: редактор видит результат до публикации.

WordPress и Omeka S предоставляют редакторы страниц, но не включают предметные экраны (timeline ректоров, flipbook) и не оптимизированы под киоск. Intuiface не имеет встроенной CMS — контент проектируется в Composer вручную.

### 4.3. Черновик, публикация и версионирование

**Преимущество.** Редактирование материалов не влияет на стенд до явной публикации; ошибочные изменения можно откатить.

**Обоснование.** Таблица `pages` хранит `draft_document` (рабочая версия) и `published_document` (версия для посетителя). Публикация копирует черновик в опубликованное поле и создаёт запись в `page_versions`. Администратор может просмотреть историю и **восстановить** предыдущую версию. При сохранении черновика действует **optimistic locking** (`documentVersion`) — конфликт параллельного редактирования возвращает HTTP 409 вместо тихой перезаписи.

Традиционный стенд: любое изменение сразу видно посетителю (замена плаката). В простых CMS (статичные HTML-киоски) откат и черновики часто отсутствуют.

### 4.4. Динамическое меню без пересборки приложения

**Преимущество.** Новый раздел экспозиции можно добавить на главный экран или в подменю через админку — без redeploy frontend.

**Обоснование.** Пункты `menu_items` загружаются через `GET /api/menu/:section`. Администратор задаёт `section`, `label`, `path`, `position`, `is_active`. Главный экран и `PathResolverPage` автоматически отображают актуальную структуру.

При традиционном подходе добавление раздела означает изготовление новых материалов и, возможно, изменение физической навигации зала. В React-приложениях без динамического меню каждый новый раздел требует правки `routes.tsx` и пересборки.

### 4.5. Каталог персоналий с ролями, поиском и медиа

**Преимущество.** Биографии ректоров, преподавателей, спортсменов ведутся в единой модели с классификацией и привязкой файлов.

**Обоснование.** `PeoplePanel` и `PersonForm` обеспечивают CRUD персоналий; назначение ролей (`rector`, `teacher-vov`, `teacher-afgan`, спортивные роли), тегов и категорий; загрузку основного и дополнительных фото; прикрепление документов. Поиск с **debounce 300 ms** и фильтры по роли/тегу/категории ускоряют работу редактора при большом каталоге.

PastPerfect ведёт «Members», но не связывает их с CMS-страницами и киоск-UI. Omeka S описывает items, но не моделирует ролевую фильтрацию для специализированных экранов стенда.

### 4.6. Файловый менеджер медиатеки

**Преимущество.** Мультимедийный архив музея организуется централизованно: upload, папки, метаданные, флаги галерей.

**Обоснование.** `FileManager` поддерживает drag-and-drop загрузку, создание папок, переименование, перемещение (DnD), удаление (soft delete), загрузку по URL, редактирование метаданных (`title`, `alt`, `year`, `showInPhotoGallery`, `showInVideoGallery`, `tags`). Изображения из медиатеки подставляются в CMS-блоки через `ImagePathInput` с поиском по имени файла.

Традиционный стенд: медиа «зашиты» в макет. Intuiface: медиа привязаны к сценам без единого реестра и без автоматического появления в фото-/видеогалереях.

## 5. Масштабируемость

### 5.1. Рост контента без изменения схемы БД

**Преимущество.** Система масштабируется по объёму экспозиционного материала — новые страницы, персоны, фото и видео — без миграций структуры данных.

**Обоснование.** CMS-контент хранится в **JSONB** (`draft_document`, `published_document`); расширение набора блоков не требует новых таблиц. Медиаметаданные — также JSONB. Добавление 100-й CMS-страницы или 500-й фотографии — операция вставки записи, а не изменение DDL.

Традиционный стенд масштабируется физически (новые витрины, площадь). Реляционные CMS с отдельной таблицей на каждый тип блока (legacy-схема проекта до canonical migration) требовали миграции при каждом новом формате контента.

### 5.2. Добавление разделов через админку и catch-all маршрутизацию

**Преимущество.** Новые тематические направления музея (спорт, студенческая жизнь) могут быть добавлены преимущественно контентом, а не кодом.

**Обоснование.** Catch-all `*` → `PathResolverPage` в сочетании с `menu_items` и CMS-страницами позволяет создавать произвольные URL-пути, включая **спорт** и **студенческую жизнь** без отдельных React-маршрутов. **Специализированный экран** — когда нужен уникальный UX (timeline ректоров, память); **CMS + menu** — для типовых экспозиций и блока `peopleCatalog`.

### 5.3. Расширение классификации персоналий через справочники

**Преимущество.** Новые категории участников экспозиции (например, новая роль в зале славы) добавляются без изменения кода страниц.

**Обоснование.** CRUD для `roles`, `tags`, `categories` в `TaxonomyPanel`. Фильтрация `GET /api/people?role=…` на стенде автоматически использует новые slug. Предустановленные роли загружаются миграцией; администратор может добавлять новые.

### 5.4. Профиль нагрузки, соответствующий контексту эксплуатации

**Преимущество.** Архитектура не избыточна для сценария «один киоск + 1–2 администратора», что упрощает развёртывание на инфраструктуре университета.

**Обоснование.** SPA без SSR, без CDN и горизонтального масштабирования — осознанный выбор для локального стенда с ограниченной аудиторией. PostgreSQL и Express справляются с профилем нагрузки музейного киоска; ресурсы не расходуются на multi-tenant SaaS-инфраструктуру.

При необходимости роста (несколько киосков в кампусе) достаточно одного backend и общей БД — клиенты независимы, stateless API.

### 5.5. Внешние видео без роста дискового пространства

**Преимущество.** Видеогалерея расширяется ссылками на YouTube/Vimeo без локального хранения файлов.

**Обоснование.** `POST /api/media/upload-url` регистрирует внешний URL; `VideoModal` воспроизводит через iframe-embed. Метаданные и теги хранятся в `media_assets.metadata`; диск сервера не заполняется копиями роликов.

## 6. Расширяемость

### 6.1. Monorepo с общим контрактом CMS-документов

**Преимущество.** Новые типы блоков и изменения формата документа синхронизируются между редактором, рендерером и API без рассогласования.

**Обоснование.** Пакет `@museum/document` определяет `PageDocument`, `BlockNode`, утилиты `walkBlocks`, `findBlockById`. Импортируется и в `@museum/web` (`BlockRenderer`, `DocumentEditor`), и в `@museum/server` (валидация JSONB). TypeScript компилятор выявляет несовместимости при рефакторинге.

Разрозненные frontend и backend (типичны для Intuiface + custom API или WordPress + custom theme) не имеют единого контракта — ошибки формата обнаруживаются только в runtime на киоске.

### 6.2. Реестр CMS-блоков как точка расширения

**Преимущество.** Добавление нового типа экспозиционного блока локализовано и не затрагивает существующие 22 типа.

**Обоснование.** Документированный паттерн: новый блок регистрируется в `cms-block-registry.ts`, добавляется view в `CmsBlockViews.tsx` и editor в `BlockEditor`. Схема PostgreSQL **не меняется** — payload хранится в JSONB `payload` блока.

### 6.3. Модульная архитектура backend

**Преимущество.** Новые доменные области API добавляются изолированно, без переписывания существующих модулей.

**Обоснование.** Слои **routes → controllers → services → Prisma**. Четыре доменных модуля: `people`, `pages`, `menu`, `media`. Новый модуль (например, «мероприятия») следует тому же шаблону: router, controller, service, Prisma-модели.

CollectionSpace и enterprise CRM монолитны; расширение требует глубокого знания платформы. Традиционный стенд не расширяется программно.

### 6.4. Стратегия «спецэкран или CMS»

**Преимущество.** Для уникального UX создаётся React-страница; для типового контента — CMS-страница в админке. Это баланс между гибкостью и скоростью наполнения.

**Обоснование.** Ректоры и память — спецэкраны с доменной логикой (timeline, flipbook, роли). Новые информационные разделы и картотеки людей — через CMS-блок `peopleCatalog` + menu без нового `.tsx`. Catch-all `PathResolverPage` обслуживает произвольные URL без новых маршрутов в `routes.tsx`.

### 6.5. REST API как основа интеграций

**Преимущество.** Внешние системы (мобильное приложение, сайт университета, другой киоск) могут потреблять те же данные через API.

**Обоснование.** Публичные эндпоинты: `/api/menu`, `/api/pages`, `/api/people`, `/api/media/gallery/*`. Headless-подход отделяет данные от представления — в отличие от monolithic WordPress theme или Intuiface-сцен, где данные «заперты» в UI.

## 7. Удобство сопровождения

### 7.1. Единый технологический стек TypeScript

**Преимущество.** Разработчик и сопровождающий работают в одной языковой среде на клиенте, сервере и shared-пакете.

**Обоснование.** Node.js 20, TypeScript ~6.0, React 19, Express 5, Prisma — единая экосистема npm. Команды `npm run dev`, `npm run build`, `npm run check` (lint + type-check) выполняются из корня monorepo через Turborepo. Не требуется параллельное сопровождение PHP-backend + JS-frontend или Java CollectionSpace + отдельный frontend.

### 7.2. Декларативная схема БД и версионированные миграции

**Преимущество.** Изменения структуры данных воспроизводимы и документированы.

**Обоснование.** `schema.prisma` — единый источник правды; `npm run db:migrate`, `db:deploy`, `db:generate` синхронизируют PostgreSQL и Prisma Client. Миграция canonical schema зафиксировала переход от legacy-таблиц к унифицированной модели — история эволюции сохранена в репозитории.

### 7.3. Конфигурация через переменные окружения

**Преимущество.** Развёртывание на сервере университета не требует правки исходного кода.

**Обоснование.** `.env.example` документирует: `PORT`, `DATABASE_URL`, `BETTER_AUTH_SECRET`, `CORS_ORIGIN`, `ADMIN_EMAIL`, `ADMIN_PASSWORD`. Пути к каталогам вычисляются от **корня monorepo** (`lib/paths.ts`), а не от `process.cwd()` — сервер корректно работает независимо от директории запуска.

### 7.4. Слоистая и компонентная структура frontend

**Преимущество.** Локализация изменений UI упрощает сопровождение при доработке стенда.

**Обоснование.** Структура `pages / layouts / patterns / cms / design-system / features / admin`. ~100+ файлов в `apps/web/src`; CMS-блоки, admin-панели и публичные patterns разделены. ESLint и Prettier обеспечивают единый стиль (`npm run lint`).

### 7.5. Отсутствие проприетарных лицензий и vendor lock-in

**Преимущество.** Сопровождение не зависит от подписки на сторонние платформы и не прерывается при изменении тарифов Intuiface, PastPerfect Online и аналогов.

**Обоснование.** Open stack: React, Express, PostgreSQL, Prisma, Better Auth — MIT/Apache/GPL-совместимые лицензии. Исходный код полностью в репозитории университета; нет обязательного облака.

### 7.6. Документированный процесс передачи проекта

**Преимущество.** Новый разработчик или администратор может развернуть систему по README без устных инструкций.

**Обоснование.** README описывает: установку (`npm install`), разработку (`npm run dev`), production-сборку (`npm run build`, `npm run start`), миграции (`db:migrate`, `db:seed-admin`), quality checks. Комплект проектной документации в `docs/` покрывает предметную область, требования, архитектуру, БД, сценарии и аналоги.

## 8. Современные технологии

### 8.1. React 19 и Vite 8 — актуальный frontend-стек

**Преимущество.** Клиентское приложение построено на поддерживаемых версиях фреймворка и сборщика с высокой скоростью разработки и оптимизированной production-сборкой.

**Обоснование.** React 19 обеспечивает компонентную модель для сложного UI (CMS-редактор, flipbook, drag-and-drop). Vite 8 — быстрый HMR в dev и **code splitting** vendor-chunks в production, снижающий объём начальной загрузки JavaScript на киоске.

Устаревшие киоски на Flash, Silverlight или статичном HTML не получают обновлений безопасности и не поддерживают современные touch-API.

### 8.2. Headless CMS на JSONB и REST API

**Преимущество.** Контент отделён от представления; тот же документ рендерится в preview админки и на публичном стенде.

**Обоснование.** PostgreSQL JSONB + `@museum/document` + `BlockRenderer` — паттерн headless CMS, применяемый в индустрии (Contentful, Strapi), но реализованный **под задачи музея** без абонентской платы SaaS. Публикация, версии, редиректы slug — на уровне `PagesService`.

### 8.3. Prisma ORM и PostgreSQL

**Преимущество.** Типобезопасный доступ к данным, транзакции, миграции — на промышленной СУБД с ACID-гарантиями.

**Обоснование.** Prisma Client генерируется из `schema.prisma`; запросы типизированы (`PersonWhereInput`, `PageUpdateInput`). Публикация CMS выполняется в `$transaction`. PostgreSQL — стандарт для production web-приложений; JSONB нативно поддерживает CMS-документы.

SQLite или файловые хранилища (типичны для простых киосков) не обеспечивают concurrent access и транзакционную целостность при параллельной работе администратора и киоска.

### 8.4. Better Auth — современная session-based аутентификация

**Преимущество.** Безопасная аутентификация администраторов без самописной логики login/logout.

**Обоснование.** Better Auth: email/password, HttpOnly session cookies, rate limiting (20 req/min на `/api/auth`), `disableSignUp: true`, проверка production-секретов. Helmet и CORS с whitelist. Пароли не хранятся в localStorage — снижен риск XSS на общедоступном киоске в той же сети.

### 8.5. Современные библиотеки для предметных сценариев

**Преимущество.** Специализированный UX реализован проверенными open source компонентами, а не самодельными решениями.

**Обоснование.**

| Задача | Технология |
|--------|------------|
| PDF-flipbook | `pdfjs-dist`, `react-pageflip`, IndexedDB-кэш |
| Анимации киоска | Framer Motion (ScreenSaver, галереи) |
| Drag-and-drop в CMS и File Manager | `@dnd-kit` |
| Видео YouTube/Vimeo | iframe-embed в `VideoModal` |
| Стилизация | Tailwind CSS 4, `@tailwindcss/vite` |

### 8.6. Turborepo и npm workspaces

**Преимущество.** Monorepo управляется инструментами, стандартизированными в индустрии (2024–2026), с кэшем сборки и явным графом зависимостей.

**Обоснование.** `@museum/server#build` зависит от `@museum/web#build` — production-сервер всегда раздаёт актуальную статику. Параллельный `dev` для Vite и Express ускоряет разработку.

## 10. Выводы

Разрабатываемая система `museum` превосходит **традиционные музейные стенды** за счёт интерактивности, мгновенной навигации, мультимедиа и централизованного управления контентом. По сравнению с **существующими цифровыми решениями** (универсальные CMS, CRM коллекций, конструкторы киосков) она обеспечивает **целостность** «стенд + админка + API», **предметную специализацию** под музей ГрГУ и **экономическую самостоятельность** (self-hosted, без абонентской платы).

Ключевые преимущества носят не абстрактный, а **прикладной характер**: они следуют из конкретных архитектурных решений и реализованной функциональности проекта — SPA-киоск с ScreenSaver, headless CMS с 21 блоком и версионированием, динамическое меню, каталог персоналий, медиатека и современный TypeScript-стек с PostgreSQL.

Ограничения системы (отсутствие полноценного фондового учёта, глобального поиска на стенде, enterprise-стандартов метаданных) осознанно выходят за рамки scope проекта и не умаляют перечисленных преимуществ для задачи **цифрового интерактивного стенда университетского музея**.
