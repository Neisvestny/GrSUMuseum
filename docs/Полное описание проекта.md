## 1. Назначение системы

### 1.1. Общая характеристика

Программный комплекс **museum** — интерактивная информационно-просветительская система для **университетского музея ГрГУ**. Система предназначена для эксплуатации на **сенсорном киоске** (touch-экран в полноэкранном браузере) в здании университета и одновременно выполняет роль **headless CMS** — платформы управления цифровым контентом экспозиции.

В отличие от классического музея с физическими витринами, проект реализует **цифровую экспозицию**: история университета, персоналии, спорт, студенческая жизнь, фото- и видеоархив, архивные PDF-документы. Тематические разделы **спорт** и **студенческая жизнь** оформляются через **CMS и динамическое меню** (`PathResolverPage`), без отдельных React-файлов на каждый URL.

### 1.2. Две функции системы

| Функция | Аудитория | Описание |
|---------|-----------|----------|
| **Публичный интерактивный стенд** | Посетители музея | Самостоятельное ознакомление с разделами, мультимедиа, биографиями, PDF-книгами |
| **Административная панель** | Сотрудники музея | Управление меню, CMS-страницами, персоналиями, медиатекой, публикация на стенд |

### 1.3. Состав monorepo

| Пакет | Путь | Назначение |
|-------|------|------------|
| `@museum/web` | `apps/web` | React SPA: киоск + admin UI |
| `@museum/server` | `apps/server` | Express REST API, auth, раздача статики |
| `@museum/document` | `packages/document` | Общие типы CMS-документов (`PageDocument`, `BlockNode`) |

### 1.4. Режим эксплуатации

- **Development:** Vite dev server (`:5173`) + Express API (`PORT` из `.env`, по умолчанию 3000), параллельно через Turborepo.
- **Production:** `npm run build` → Express раздаёт собранный SPA из `apps/web/dist/`, медиа из `apps/web/public/`, обрабатывает `/api/*`; catch-all fallback на `index.html` для client-side маршрутов.

## 2. Бизнес-цели

### 2.1. Стратегические цели

1. **Цифровизация экспозиции** — замена статичных плакатов и разрозненных материалов единой интерактивной платформой.
2. **Актуальность контента** — сотрудники музея обновляют материалы через admin-панель без программиста при типовых изменениях.
3. **Представление наследия ГрГУ** — история, ректоры, память (ВОВ, Афганистан), спорт, студенческая жизнь, мультимедиа.
4. **Киоск-UX** — удобство для посетителей выставочного зала: крупные кнопки, жесты, автосброс сеанса.
5. **Технологическая автономия** — open stack (TypeScript, PostgreSQL), self-hosted на инфраструктуре университета, без абонентских SaaS.

### 2.2. Решаемые проблемы

| Традиционный подход | Проблема | Решение в `museum` |
|---------------------|----------|-------------------|
| Бумажные каталоги | Сложно обновлять, нет мультимедиа | БД + REST API + admin |
| Статичный HTML-сайт | Каждое изменение = правка кода | Headless CMS, JSONB, dynamic menu |
| PowerPoint на экране | Плохой touch-UX | SPA, меню, ScreenSaver, жесты |
| Проприетарные CMS | Стоимость, vendor lock-in | Open source стек |
| Excel + папки с файлами | Нет связей между сущностями | Единая реляционная модель |
| «Застрявший» киоск | Чужая сессия следующему посетителю | ScreenSaver 5 мин: переход на `/` + заставка; клик снимает оверлей |

## 3. Функциональные возможности

### 3.1. Публичный стенд (посетители)

#### Навигация и оболочка

- Главный экран с кнопками разделов из `GET /api/menu/home` (320×144 px).
- Подменю секций через `PathResolverPage` + `GET /api/menu/:section`.
- Catch-all маршрут `*` → подменю или CMS-страница.
- Возврат назад: кнопка в navbar, кнопка на подменю, **свайп вправо** ≥ 80 px.
- **ScreenSaver** после 5 мин бездействия: переход на `/` и показ заставки; клик снимает оверлей (не работает на `/admin*`).
- Состояния: Loading, Error, Empty — на всех data-driven страницах.

#### CMS-экспозиции

- Просмотр опубликованных страниц по URL (`GET /api/pages/by-path`).
- Рендеринг **22 типов блоков** через `BlockRenderer` (в т.ч. `peopleCatalog` — картотека людей).
- Редirect slug при переименовании (`page_redirects`).

#### Специализированные экраны

| Маршрут | Функция |
|---------|---------|
| `/history/rectors` | Хронология ректоров (timeline, спецэкран `Rectors`) |
| `/history/rectors/:id` | Карточка ректора (детальный маршрут), документы |
| `/history/memory/vov` | «Купаловцы помнят» — ВОВ |
| `/history/memory/afgan` | «Купаловцы помнят» — Афганистан |
| `/gallery` | Фотогалерея по годам, lightbox |
| `/video-gallery` | Видеогалерея, фильтр по тегам, modal |

**Через CMS и catch-all `*` (без отдельных `.tsx`-маршрутов):** `/sport`, `/sport/*`, `/studentlife`, `/studentlife/*` и любые другие пути — подменю (`GET /api/menu/:section`) или CMS-страница по slug.

#### Память и PDF

- Вкладки: преподаватели (`EntityListDetail`) + PDF flipbook (`pdfjs-dist`, `react-pageflip`).
- Отдельные PDF: `/book_vov.pdf` (ВОВ), `/book_afgan.pdf` (Афганистан).
- IndexedDB-кэш страниц PDF (ключ — путь файла) для повторного открытия.

#### Галереи

- Фото: `showInPhotoGallery`, группировка по `metadata.year`, lazy load.
- Видео: локальные файлы + YouTube/Vimeo embed; фильтр по тегам.

### 3.2. Административная панель

- Аутентификация Better Auth (`/admin/login`).
- Пять модулей: **Меню**, **CMS страницы**, **Люди**, **Справочники**, **Медиа**.
- CMS: блочный редактор, **preview** (`BlockRenderer` с режимом `preview`), черновик, публикация, версии.
- Удаление CMS-страницы — только после **подтверждения** (title, slug, предупреждение о soft delete).
- People: CRUD, роли/теги/категории, поиск с debounce 300 ms.
- Media: File Manager (upload, DnD, папки, metadata, gallery flags).
- Toast-уведомления об операциях.

### 3.3. Что не реализовано

- Глобальный поиск на публичном стенде.
- RBAC (editor/moderator/super_admin).
- UI reorder для людей и галерей (API есть).
- SSR / SEO, многоязычность UI.
- Offline/PWA, Docker, Nginx в репозитории.
- Фондовый учёт музейных предметов (CDWA/LIDO).

### 3.4. Типы CMS-блоков (22)

| Группа | Типы |
|--------|------|
| Структура | `tabs`, `tab` |
| Текст и медиа | `hero`, `heading`, `richText`, `textImage`, `alternating`, `quote`, `callout`, `list`, `twoColumns` |
| Галереи и видео | `mediaStrip`, `imageGallery`, `video`, `embed` |
| Секции | `stats`, `features`, `accordion`, `timeline`, `buttonRow` |
| Персоналии | `peopleCatalog` — картотека: поиск по ФИО, список слева, карточка справа; роль задаётся в payload |
| Прочее | `divider` |

Блок `peopleCatalog`: payload `{ role, searchPlaceholder?, emptyText? }`. При пустом списке людей раскладка сохраняется, показывается сообщение из `emptyText`. В admin-preview — компактный режим (`CmsPreviewContext`).

Структура блока: `{ id, type, schemaVersion, payload, children[] }`. Документ: `{ blocks: BlockNode[] }`.

## 4. Пользовательские роли

### 4.1. Действующие лица

| Роль | Аутентификация | Интерфейс | Полномочия |
|------|----------------|-----------|------------|
| **Посетитель** | Нет | Публичный стенд | Просмотр опубликованного контента |
| **Редактор контента** | Email + пароль | `/admin` | Полный доступ: меню, CMS, люди, медиа, публикация |
| **Администратор системы** | То же | `/admin` + CLI | Создание учётных записей через `db:seed-admin`; единый уровень прав с редактором |
| **Система** | — | Автоматика | ScreenSaver, PathResolver, валидация, soft delete |

### 4.2. Модель доступа

- **Публичная часть:** все GET API для pages, menu, people, media/gallery — без auth.
- **Admin:** mutating API + `/admin/*` — только с session cookie Better Auth.
- **Регистрация:** `disableSignUp: true`; пароль ≥ 12 символов.
- **Rate limit:** 20 req/min на `/api/auth`.

## 5. Архитектура

### 5.1. Общая модель

**Двухtier full-stack SPA + REST API** в monorepo TypeScript.

```
┌─────────────────────────────────────────┐
│  Браузер (киоск / admin)                │
│  React 19 SPA (@museum/web)             │
└──────────────────┬──────────────────────┘
                   │ HTTP JSON, credentials:include
┌──────────────────▼──────────────────────┐
│  Express 5 (@museum/server)               │
│  /api/auth  /api/people|pages|menu|media  │
│  static: /images /videos /files + dist    │
└──────────────────┬──────────────────────┘
                   │ Prisma
┌──────────────────▼──────────────────────┐
│  PostgreSQL                               │
└───────────────────────────────────────────┘
         + apps/web/public/ (файлы на диске)
```

### 5.2. Backend: слоистая архитектура

```
routes → controllers → services → Prisma → PostgreSQL
```

Доменные модули: `people`, `pages`, `menu` — в `modules/`; `media` — `routes/media.router.ts`; `auth` — `auth/`.

Middleware: CORS, helmet, express.json, rate limit (auth), `requireAuth`, errorHandler.

### 5.3. Frontend: структура

```
pages/          — экраны маршрутов
layouts/        — MainLayout, navbar
patterns/       — SectionMenuPage, EntityListDetail, PeopleCatalog
cms/            — BlockRenderer, CmsBlockViews
features/admin/ — панели admin
design-system/  — Button, Card, States, BaseModal
hooks/          — useMenuSection, usePageByPath, usePerson
api/            — клиентские функции REST
lib/            — cms-block-registry, auth-client
```

### 5.4. Headless CMS

- **Черновик** (`draft_document`) — редактирование в admin.
- **Публикация** — копия в `published_document` + запись в `page_versions`.
- **Optimistic locking** — `documentVersion`, HTTP 409 при конфликте.
- **Контракт** — `@museum/document` shared между web и server.

### 5.5. Стратегия маршрутизации

1. **Фиксированные React-маршруты** — ректоры (timeline), память, галереи.
2. **Catch-all `*`** — `PathResolverPage`: menu section → подменю; иначе CMS-страница.
3. **Спорт, студенческая жизнь и прочие темы** — только через меню + CMS (без `pages/sport/`, `pages/studentlife/`).

### 5.6. Хранение медиа

- **Бинарные файлы:** `apps/web/public/images|videos|files/`.
- **Метаданные:** `media_assets` + JSONB (`showInPhotoGallery`, `year`, `tags`, …).
- **Сервис:** `MediaStorageService` синхронизирует FS и БД.

### 5.7. Профиль нагрузки

Один киоск + 1–2 администратора. Горизонтальное масштабирование не реализовано.

## 7. Структура базы данных

### 7.1. Обзор

**19 таблиц** PostgreSQL, управляемых Prisma (`schema.prisma`).

| Область | Таблицы |
|---------|---------|
| Auth (Better Auth) | `user`, `session`, `account`, `verification` |
| CMS | `pages`, `page_versions`, `page_redirects` |
| Персоналии | `people`, `roles`, `tags`, `categories`, `person_roles`, `person_tags`, `person_categories`, `person_media`, `person_documents` |
| Медиа | `media_folders`, `media_assets` |
| Навигация | `menu_items` |

### 7.2. Ключевые сущности

#### `pages` — CMS-страница

| Поле | Назначение |
|------|------------|
| `slug` | URL-путь (уникальный) |
| `title` | Заголовок |
| `draft_document` | JSONB черновик |
| `published_document` | JSONB для стенда (nullable) |
| `document_version` | Версия для optimistic lock |
| `deleted_at` | Soft delete |

#### `people` — персоналия

ФИО, годы, описания, `img`, `sort_order`. Связи M:N с roles, tags, categories. Доп. фото (`person_media`), документы (`person_documents`).

#### `media_assets` — медиафайл

`src`, `mime_type`, `title`, `alt`, `metadata` (JSONB: `showInPhotoGallery`, `showInVideoGallery`, `year`, `tags`, `position`, …).

#### `menu_items` — пункт меню

`section`, `label`, `path` (unique), `position`, `is_active`, `parent_id`.

### 7.3. Предустановленные роли (seed)

`rector`, `teacher-vov`, `teacher-afgan`, `olympic-coach`, `olympic-student`, `trainer`.

### 7.4. Принципы хранения

- **JSONB** для CMS и metadata — расширение без DDL.
- **Soft delete** — `pages`, `people`, `media_assets`.
- **Junction tables** — M:N для roles/tags/categories.
- **Файлы на диске** — URL в `src`; Express static.

### 7.5. Полный перечень таблиц

#### Аутентификация

| Таблица | Назначение |
|---------|------------|
| `user` | Учётная запись admin (id, name, email, email_verified, image) |
| `session` | Сессия (token, expires_at, user_id, ip, user_agent) |
| `account` | Credentials провайдера (password hash, OAuth tokens) |
| `verification` | Одноразовые коды Better Auth |

#### CMS

| Таблица | Назначение |
|---------|------------|
| `pages` | CMS-страница: slug, title, theme_key, sidebar_enabled, draft/published JSONB, document_version, updated_by, deleted_at |
| `page_versions` | Снимок document при каждой публикации |
| `page_redirects` | from_slug → to_slug при переименовании |

#### Персоналии

| Таблица | Назначение |
|---------|------------|
| `people` | ФИО, годы, описания, img, sort_order, deleted_at |
| `roles` | Справочник ролей (slug, label, sort_order) |
| `tags` | Справочник тегов |
| `categories` | Справочник категорий |
| `person_roles` | M:N person ↔ role |
| `person_tags` | M:N person ↔ tag |
| `person_categories` | M:N person ↔ category |
| `person_media` | M:N person ↔ доп. фото |
| `person_documents` | Документ персоны (title + media_asset_id) |

#### Медиа и меню

| Таблица | Назначение |
|---------|------------|
| `media_folders` | Дерево папок (parent_id) |
| `media_assets` | src, mime_type, metadata JSONB, folder_id, deleted_at |
| `menu_items` | section, label, path, position, is_active, parent_id |

### 7.6. Жизненный цикл данных CMS

1. **Create** — пустой `{ blocks: [] }` в draft; published = null.
2. **Edit** — PATCH draft, documentVersion++.
3. **Publish** — draft → published; INSERT page_versions (transaction).
4. **Restore** — version → draft.
5. **Slug change** — INSERT page_redirects.
6. **Delete** — soft delete deleted_at.

13. **UC-18–19** Вход / выход.

16. **UC-24–26** Люди и справочники.

19. **UC-30** Error/empty states.

### 9.4. Детализация ключевых сценариев

#### UC-06. Просмотр CMS-страницы

**Предусловия:** опубликованная страница для URL; slug разрешён через redirects.  
**Поток:** PathResolver → `GET /api/pages/by-path` → `CmsPageContent` → `BlockRenderer` → интерактивные блоки (tabs, accordion, buttonRow).  
**Альтернативы:** loading; empty «страница не настроена»; error state.  
**Результат:** посетитель ознакомился с экспозицией.

#### UC-21–22. Редактирование и публикация CMS

**Поток:** выбор страницы → DocumentEditor (DnD blocks) → preview → save draft (PATCH + version) → publish (POST) → toast «Опубликовано».  
**Альтернативы:** isDirty warning; HTTP 409 version conflict; publish without save confirm.  
**Результат:** контент на стенде обновлён.

#### UC-10. PDF flipbook

**Поток:** вкладка «Книга» → load `/book_vov.pdf` → pdfjs render → react-pageflip → IndexedDB cache.  
**Альтернативы:** progress «N/M стр.»; error «Не удалось загрузить PDF».  
**Результат:** архивный документ просмотрен.

## 13. Ограничения

### 13.1. Технические

- Нет global search на стенде.
- Нет offline/PWA.
- Нет SSR/SEO.
- Upload limit 50 MB.
- Нет фондового учёта экспонатов.
- Новый CMS-блок = код (registry + view + editor).
- Hardcoded: ScreenSaver 5 min, PDF paths (`book_vov.pdf`, `book_afgan.pdf`).
- Нет автотестов.
- Reorder API без UI.

### 13.1.1. Production SPA

- Catch-all в `bootstrap.ts` отдаёт `index.html` для client-side маршрутов (кроме `/api/*` и статических файлов из `dist`).

### 13.2. Организационные

- Единый admin-доступ (нет RBAC).
- Seed-only создание пользователей.
- Зависимость от разработчика для spec-экранов.
- Нет workflow approval.
- Single-tenant (только ГрГУ).
- Backup не в репозитории.

### 13.3. Инфраструктурные

- Нет Docker/Nginx в проекте.
- Single Node + single PostgreSQL.
- Медиа на local disk.
- Зависимость от LAN.

### 13.4. Пользовательские

- Limited a11y на киоске.
- Admin не для mobile.
- Нет i18n.
- Нет персонализации посетителя.
- Требуется цифровая грамотность редактора.

### 13.5. Матрица ограничений (28 пунктов)

Детальная классификация: **ОГР-Т-01…10** (технические), **ОГР-О-01…06** (организационные), **ОГР-И-01…06** (инфраструктурные), **ОГР-П-01…08** (пользовательские). Наиболее критичны для эксплуатации: отсутствие automated backup (ОГР-О-06), single point of failure (ОГР-И-03), отсутствие global search (ОГР-Т-01).
