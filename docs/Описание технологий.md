# Описание технологий

## Программный комплекс интерактивного музейного стенда ГрГУ

**Проект:** `museum` (monorepo full-stack TypeScript)  
**Версия документа:** 1.0  
**Основание:** `package.json`, `README.md`, конфигурационные файлы репозитория  
**Node.js:** 20 (`.nvmrc`)

---

## Сводная схема технологического стека

```
┌─────────────────────────────────────────────────────────────┐
│  Клиент (киоск / браузер)                                    │
│  React 19 + TypeScript + Vite 8 + React Router 7 + Tailwind 4│
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTP /api, static /images|videos|files
┌──────────────────────────▼──────────────────────────────────┐
│  Сервер                                                      │
│  Node.js 20 + Express 5 + Better Auth + Prisma 7 + multer    │
└──────────────────────────┬──────────────────────────────────┘
                           │ SQL
┌──────────────────────────▼──────────────────────────────────┐
│  PostgreSQL                                                  │
└─────────────────────────────────────────────────────────────┘
```

**Примечание.** В репозитории отсутствуют конфигурации Docker, Nginx и Cloudflare Tunnel; развёртывание описано как запуск Node.js-процесса с раздачей статики Express (`README.md`). Файл `PRODUCT_DOCS.md` в корне репозитория описывает иной проект (сайт кафедры на Next.js/NestJS) и **не отражает** технологический стек `museum`.

## Frontend

### React

**Статус в проекте:** используется (версия ^19.2.5, пакет `@museum/web`).

#### Назначение

React — JavaScript-библиотека для построения пользовательских интерфейсов на основе компонентной модели и декларативного описания UI. Является фундаментом клиентской части музейного стенда.

#### Основные возможности

- компонентная архитектура с переиспользуемыми UI-блоками;
- виртуальный DOM и reconciliation для эффективного обновления интерфейса;
- хуки (`useState`, `useEffect`, `useCallback`, `useMemo`, `useRef`) для управления состоянием и побочными эффектами;
- JSX-синтаксис для совмещения разметки и логики;
- экосистема библиотек (маршрутизация, анимации, DnD).

#### Роль в проекте

React реализует весь клиентский интерфейс `@museum/web`:

- **публичный стенд** — главное меню, CMS-страницы (`BlockRenderer`), timeline ректоров, PDF-flipbook, фото- и видеогалереи, ScreenSaver;
- **админ-панель** — формы CMS, файловый менеджер, управление людьми и меню;
- **маршрутизация** через React Router DOM 7 (`App.tsx`, `routes.tsx`);
- **компонентная структура:** `pages/`, `layouts/`, `components/design-system/`, `components/cms/`, `components/features/admin/`.

Приложение работает как **SPA (Single Page Application)**: сервер отдаёт `index.html`, дальнейший рендеринг выполняется на клиенте.

#### Преимущества использования

- единый язык (TypeScript) и модель компонентов для киоска и админки;
- богатая экосистема для touch-интерфейса (Framer Motion, `@dnd-kit`, `react-pageflip`);
- декларативный UI упрощает рендеринг 22 типов CMS-блоков из JSONB-документа;
- React 19 обеспечивает актуальную версию runtime с улучшенной производительностью.

### Next.js

**Статус в проекте:** **не используется**.

#### Назначение

Next.js — full-stack React-фреймворк с поддержкой серверного рендеринга (SSR), статической генерации (SSG), файловой маршрутизации App Router и встроенных API Routes.

#### Основные возможности

- Server Components и SSR/SSG;
- файловая маршрутизация (`app/`, `pages/`);
- оптимизация изображений (`next/image`);
- ISR (Incremental Static Regeneration);
- API Routes и middleware.

#### Роль в проекте

Next.js **не применяется**. Вместо него клиентская часть собрана на **Vite 8** + **React Router DOM 7**. Backend вынесен в отдельное приложение `@museum/server` (Express), а не в API Routes Next.js.

Выбор SPA обусловлен профилем системы:

- стенд работает как **локальный киоск** в браузере; SEO не является приоритетом;
- требуются клиентские библиотеки (PDF.js, pageflip, Framer Motion, DnD CMS-редактора);
- production-сборка раздаётся Express-сервером из `apps/web/dist`.

#### Преимущества использования (в контексте альтернативы)

Отказ от Next.js для данного проекта оправдан упрощением стека: один frontend-сборщик (Vite), явное разделение client/server в monorepo, отсутствие необходимости в SSR для музейного киоска.

### TypeScript

**Статус в проекте:** используется (~6.0.2) во всём monorepo.

#### Назначение

TypeScript — типизированный надмножество JavaScript, компилируемое в JavaScript. Обеспечивает статическую проверку типов на этапе разработки и сборки.

#### Основные возможности

- статическая типизация переменных, функций, классов и интерфейсов;
- generics, union/intersection types, type inference;
- интеграция с IDE (автодополнение, рефакторинг);
- постепенное внедрение (strict mode, `tsc --noEmit`);
- декларации типов для сторонних библиотек (`@types/*`).

#### Роль в проекте

TypeScript применяется во всех пакетах monorepo:

| Пакет | Назначение типов |
|-------|------------------|
| `@museum/web` | props компонентов, hooks, API-клиент, CMS registry |
| `@museum/server` | controllers, services, Prisma models, env |
| `@museum/document` | `PageDocument`, `BlockNode` — контракт CMS |

Команда `npm run type-check` (Turborepo) выполняет проверку типов без emit. ESLint с `@typescript-eslint` дополняет статический анализ.

#### Преимущества использования

- единый контракт CMS между редактором, API и рендерером через `@museum/document`;
- раннее обнаружение ошибок в 22 типах CMS-блоков и REST API;
- улучшенная сопровождаемость monorepo при курсовой/дипломной разработке;
- типобезопасная работа с Prisma Client (генерируемые типы).

## Backend

### NestJS

**Статус в проекте:** **не используется**.

#### Назначение

NestJS — прогрессивный Node.js-фреймворк для построения серверных приложений с архитектурой, inspired by Angular: модули, dependency injection, декораторы, guards, pipes.

#### Основные возможности

- модульная архитектура (`@Module`, `@Controller`, `@Injectable`);
- встроенная поддержка TypeScript;
- интеграция с ORM (Prisma, TypeORM), очередями (BullMQ), WebSocket;
- guards, interceptors, filters для cross-cutting concerns;
- OpenAPI/Swagger из коробки.

#### Роль в проекте

NestJS **не применяется**. Backend реализован на **Express 5** (`apps/server/src/app/create-app.ts`) с ручной регистрацией маршрутов:

```
routes → controllers → services → Prisma
```

Middleware: `cors`, `helmet`, `express-rate-limit`, `requireAuth`, `errorHandler`.

#### Преимущества использования (в контексте альтернативы)

Express выбран как **минималистичный HTTP-сервер** без overhead DI-контейнера NestJS. Для музейного стенда с ограниченным API (`/people`, `/pages`, `/menu`, `/media`, `/auth`) этого достаточно; monorepo уже обеспечивает модульность через файловую структуру `modules/`, `services/`, `controllers/`.

---

### Node.js

**Статус в проекте:** используется (версия ≥ 20, `.nvmrc`: `20`).

#### Назначение

Node.js — серверная среда выполнения JavaScript/TypeScript на движке V8. Обеспечивает event-driven модель для I/O-интенсивных веб-приложений.

#### Основные возможности

- неблокирующий I/O и event loop;
- npm/ecosystem пакетов;
- native ESM (`"type": "module"` в `package.json`);
- HTTP-сервер из коробки;
- файловая система, streams, crypto.

#### Роль в проекте

Node.js — runtime для:

- **Express-сервера** (`apps/server`) — REST API, Better Auth, раздача статики;
- **инструментов разработки** — Vite dev server, `tsx watch`, Prisma CLI, ESLint;
- **скриптов** — `seed-admin.ts`, `migrate-legacy-data.ts`, `verify-paths.ts`.

Production-запуск: `node dist/index.js` после `npm run build`. Turborepo orchestrates параллельную сборку пакетов.

#### Преимущества использования

- единый язык TypeScript на frontend и backend (full-stack monorepo);
- асинхронная модель подходит для REST API с PostgreSQL и загрузкой файлов (multer);
- LTS-версия 20 обеспечивает стабильность для развёртывания на сервере университета;
- широкая экосистема npm-пакетов (Express, Prisma, Better Auth).

## База данных

### PostgreSQL

**Статус в проекте:** используется (provider в `schema.prisma`: `postgresql`).

#### Назначение

PostgreSQL — объектно-реляционная СУБД с поддержкой ACID-транзакций, расширенных типов данных и JSONB. Является центральным хранилищем данных музейного стенда.

#### Основные возможности

- реляционная модель с foreign keys и каскадным удалением;
- тип **JSONB** для полуструктурированных документов CMS;
- транзакции (`BEGIN`/`COMMIT`), `$transaction` в Prisma;
- индексы, unique constraints;
- полнотекстовый поиск (в проекте для людей используется `contains` с `mode: 'insensitive'`).

#### Роль в проекте

PostgreSQL хранит:

| Группа таблиц | Содержимое |
|---------------|------------|
| Auth (Better Auth) | `user`, `session`, `account`, `verification` |
| CMS | `pages` (draft/published JSONB), `page_versions`, `page_redirects` |
| Контент | `people`, роли/теги/категории, `menu_items` |
| Медиа | `media_assets`, `media_folders`, связи с людьми |

Подключение через `DATABASE_URL` или отдельные `DB_*` переменные (`.env.example`). Миграции — Prisma Migrate (`npm run db:migrate`, `db:deploy`).

#### Преимущества использования

- JSONB идеален для **блочной CMS** (22 типа блоков без ALTER TABLE при добавлении типа);
- транзакционность при публикации страниц и reorder персоналий;
- надёжность и зрелость для долгосрочного хранения музейного контента;
- совместимость с Prisma ORM и `@prisma/adapter-pg`.

## Инфраструктура

### Docker

**Статус в проекте:** **не используется** (отсутствуют `Dockerfile`, `docker-compose.yml`).

#### Назначение

Docker — платформа контейнеризации для упаковки приложений с зависимостями в изолированные образы, воспроизводимые на любой среде.

#### Основные возможности

- образы (images) и контейнеры;
- Docker Compose для multi-container стеков;
- volume для персистентных данных;
- изоляция окружения.

#### Роль в проекте

Контейнеризация **не реализована**. Развёртывание по `README.md`:

```bash
npm run build
npm run start
```

Приложение запускается как Node.js-процесс; PostgreSQL устанавливается отдельно. Медиафайлы хранятся в файловой системе `apps/web/public/`.

#### Преимущества использования (перспектива)

Docker мог бы упростить развёртывание на сервере ГрГУ (единый образ app + postgres), но текущая архитектура ориентирована на **прямую установку** Node.js 20 и PostgreSQL — что соответствует учебному/курсовому профилю проекта.

---

### Nginx

**Статус в проекте:** **не используется** (отсутствуют конфигурационные файлы).

#### Назначение

Nginx — высокопроизводительный веб-сервер и reverse proxy. Используется для терминирования TLS, балансировки нагрузки, раздачи статики и проксирования к application server.

#### Основные возможности

- reverse proxy к Node.js/Express;
- раздача static files с кэшированием;
- SSL/TLS termination;
- rate limiting, gzip, load balancing.

#### Роль в проекте

Nginx **не настроен** в репозитории. В production Express **самостоятельно**:

- раздаёт API (`/api/*`);
- отдаёт статику SPA (`apps/web/dist`);
- раздаёт медиа (`apps/web/public/` → `/images`, `/videos`, `/files`).

В development Vite proxy перенаправляет `/api` на Express (`vite.config.ts`).

#### Преимущества использования (перспектива)

Nginx мог бы вынести раздачу статики и TLS на edge-слой, но для **локального киоска** в сети университета достаточно одного Express-процесса — профиль нагрузки минимален.

---

### Cloudflare Tunnel

**Статус в проекте:** **не используется**.

#### Назначение

Cloudflare Tunnel (`cloudflared`) — сервис безопасного публикации локальных серверов в интернет через сеть Cloudflare без открытия входящих портов и публичного IP.

#### Основные возможности

- исходящее соединение к Cloudflare Edge;
- защита от DDoS;
- доступ к localhost-сервисам через публичный домен;
- не требует проброса портов на роутере.

#### Роль в проекте

Cloudflare Tunnel **не применяется**. Музейный стенд проектируется для **on-premise** эксплуатации в здании ГрГУ (локальная сеть или localhost на киоске), без обязательного выхода в публичный интернет.

#### Преимущества использования (перспектива)

Туннель был бы актуален при удалённом администрировании или демонстрации стенда извне; для текущей реализации не требуется.

## Аутентификация

### Better Auth

**Статус в проекте:** используется (версия ^1.6.11, `@better-auth/prisma-adapter`).

#### Назначение

Better Auth — TypeScript-библиотека аутентификации для full-stack приложений. Предоставляет готовые endpoints, session management и адаптеры к ORM.

#### Основные возможности

- email/password authentication;
- session-based auth с cookie;
- Prisma adapter для хранения users/sessions;
- настраиваемые cookie attributes (`httpOnly`, `sameSite`, `secure`);
- отключение публичной регистрации (`disableSignUp`);
- интеграция с Express через `toNodeHandler` (`better-auth/node`).

#### Роль в проекте

Better Auth обеспечивает **единственный контур аутентификации** — доступ администраторов к `/admin`:

| Компонент | Файл / конфигурация |
|-----------|---------------------|
| Server config | `apps/server/src/auth/auth.ts` |
| Express mount | `app.all('/api/auth/*splat', toNodeHandler(auth))` |
| Client | `apps/web/src/lib/auth-client.ts`, `useSession()` |
| UI | `AdminLogin.tsx`, `ProtectedRoute.tsx` |
| Seed admin | `scripts/seed-admin.ts` |

Параметры: `minPasswordLength: 12`, `disableSignUp: true`, `BETTER_AUTH_SECRET` обязателен в production (`assertProductionSecrets`).

#### Преимущества использования

- готовое решение без написания JWT/refresh logic с нуля;
- нативная интеграция с Prisma и PostgreSQL (таблицы `user`, `session`, `account`);
- TypeScript-first API согласован с monorepo;
- session cookie безопаснее localStorage для admin-панели на том же origin, что и API.

## ORM

### Prisma

**Статус в проекте:** используется (версия ^7.8.0, `@prisma/client`, `@prisma/adapter-pg`).

**Альтернатива Drizzle ORM:** **не используется**. В зависимостях проекта отсутствует `drizzle-orm`; Better Auth подключён через `@better-auth/prisma-adapter`, а не drizzle-adapter.

#### Назначение

Prisma — современная ORM для Node.js/TypeScript с декларативной схемой данных, type-safe клиентом и системой миграций.

#### Основные возможности

- декларативная схема (`schema.prisma`);
- автогенерация типобезопасного `PrismaClient`;
- Prisma Migrate — версионирование DDL;
- поддержка JSON, relations, soft delete patterns;
- `@prisma/adapter-pg` для драйвера `pg`;
- `$transaction` для атомарных операций.

#### Роль в проекте

Prisma — **единственный слой доступа к данным** в `@museum/server`:

```
apps/server/prisma/schema.prisma  →  generated client  →  services  →  controllers
```

Ключевые модели: `Page`, `PageVersion`, `Person`, `MenuItem`, `MediaAsset`, `User`, `Session`. CMS-документы хранятся в полях `Json` (`draftDocument`, `publishedDocument`).

Команды: `db:generate`, `db:migrate`, `db:deploy`, `db:seed-admin`.

#### Преимущества использования

- миграции воспроизводят схему БД на сервере университета;
- автогенерация типов исключает расхождение SQL и TypeScript;
- JSONB-поля CMS типизируются на уровне приложения через `@museum/document`;
- официальный adapter Better Auth для Prisma снижает объём auth-кода.

---

### Drizzle ORM

**Статус в проекте:** **не используется**.

#### Назначение

Drizzle ORM — лёгкая TypeScript ORM с SQL-like синтаксисом запросов и schema-as-code подходом.

#### Основные возможности

- type-safe queries близкие к SQL;
- drizzle-kit для миграций;
- минимальный runtime overhead;
- адаптер Better Auth (`@better-auth/drizzle-adapter`).

#### Роль в проекте

Drizzle **не применяется**. Выбран **Prisma** (см. раздел выше) как ORM с более зрелой экосистемой миграций и генерацией клиента, уже интегрированным в проект.

#### Преимущества использования (в контексте выбора Prisma)

Drizzle мог бы дать меньший bundle на server, но для музейного стенда с PostgreSQL и Better Auth Prisma обеспечивает более прямую интеграцию (`prismaAdapter`, `@prisma/adapter-pg`) без дополнительной настройки.
